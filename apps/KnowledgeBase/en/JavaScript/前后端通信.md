# Ajax

**AJAX** (Asynchronous JavaScript and XML), a technology for exchanging data with server and updating partial webpage content without refreshing entire page;

**Long Polling**: A real-time communication mode implemented based on AJAX technology; client sends request to server via AJAX, after server receives request, doesn't immediately return response, but keeps connection for a period of time; if during this period server has new data to push, immediately responds with data, otherwise responds with empty data, simultaneously closes current connection; after client receives response (whether empty or data response), immediately sends new AJAX request, forming wait-response-wait again polling loop;

**Short Polling**: A communication mode based on AJAX for client to actively get real-time data from server; client sends AJAX request to server every fixed time (like 1 second, 5 seconds) (e.g., using setInterval to send), server immediately returns response after receiving request (whether there's new data or not), client processes response, then waits for next interval to repeat request; Disadvantages: Data update delay depends on polling interval. For example, if interval is 5 seconds, if new message is generated at 3rd second, client has to wait until 5th second to get it, maximum 5 seconds delay; most of the time server may not have new data, but client still sends requests at intervals, wasting network bandwidth and server resources;

### Basic Principle

Creates `XMLHttpRequest` object via JS (or uses `fetch`), configures request info (like request method GET/POST, request URL, whether async, etc.), sends async request to server, listens for server's response status (like success, failure, timeout, etc.), receives data returned by server (JSON/XML), parses data with JS, updates page partial content via DOM operations;

AJAX requests are subject to browser same-origin policy, can only request resources from same domain, same port, same protocol; cross-origin requests require server to configure CORS;

### Network Request Methods

Network request methods are rules and patterns for exchanging data between client and server, core based on network protocols (like HTTP, HTTPS, WebSocket, etc.), and implement data sending, getting, updating, etc. through different methods or strategies;

**GET**: Gets specified resource from server, won't modify server data; request parameters attached in URL (like `https://api.example.com/users?id=1`), parameters have length limit (usually 2KB~8KB, depends on browser), browser caches GET request responses (can control via `Cache-Control`);

**POST**: Submits data to server, usually used to create new resource data; request parameters placed in request body (Body), no length limit, can transmit large amounts of data (like JSON, form data, files); no default cache, each request triggers server processing;

**PUT**: Sends data to server, used for full update of specified resource (creates if resource doesn't exist); parameters usually in request body, has idempotency (same PUT request executed multiple times has consistent result);

**DELETE**: Requests server to delete specified resource; parameters can be in URL or request body, has idempotency (deleting same resource multiple times has consistent result);

**PATCH**: Used for partial update of server resource; parameters can be in URL or request body, not necessarily idempotent (multiple partial updates may result in different resource results);

**HEAD**: Used for client to get server's metadata about a resource without transmitting actual content (like whether resource exists, size, last modified time, etc.), similar to GET but only returns response headers, no response body;

**OPTIONS**: Used to query communication methods supported by server, lets client understand allowed HTTP request methods, cross-origin rules, supported request header formats, etc. before sending actual request;

**CONNECT**: Establishes tunnel connection (like for SSL handshake in HTTPS);

### XMLHttpRequest

Used for async HTTP communication with server without refreshing page; based on event callbacks;

```js
const xhr = new XMLHttpRequest();

xhr.open('GET', 'http://127.0.0.1', true);

xhr.onload = function () {
  if (xhr.status === 200) {
    console.log('Request successful', xhr.responseText);
  }
};

xhr.send();
```

**xhr instance**

status: HTTP status code returned by server (like 200 means success, 404 means resource not found, 500 means server error);

statusText: Text description corresponding to HTTP status code;

responseText: Response data in text format returned by server (suitable for JSON, HTML, text, etc.);

responseXML: Response data in XML format returned by server (requires server to return `Content-Type: text/xml`);

timeout: Sets request timeout (milliseconds), triggers `ontimeout` event after timeout;

withCredentials: Sets whether to allow request to carry credentials like cookies;

open(): Configures request parameters but doesn't send request; first parameter is request method, second is request URL, third is whether to send request asynchronously;

```js
xhr.open('GET', 'http://127.0.0.1', true);
```

send(): Initiates request to server; parameter is request body data (GET method doesn't need to pass, POST can pass string, `FormData`, etc.);

setRequestHeader(): Sets request headers (like `Content-Type`, `Authorization`), must be called after `open()` and before `send()`;

```js
xhr.setRequestHeader('Authorization', 'Bearer abc123');
```

abort(): Interrupts current request;

getResponseHeader(): Gets specified response header returned by server;

```js
xhr.getResponseHeader('content-type');
```

onload: Callback triggered after request succeeds;

```js
xhr.onload = function () {
  if (xhr.status === 200) {
    console.log('Request successful', xhr.responseText);
  }
};
```

onerror: Callback triggered when request fails; HTTP errors don't trigger;

onabort: Triggered when `abort()` is called to interrupt request;

onprogress: Continuously triggered while receiving response body;

ontimeout: Triggered when request times out;

### Fetch

Promise-based network request API; first parameter is URL, second is request config object, returns a Promise object; returned Promise only rejects on network error, otherwise resolves (even HTTP 4xx/5xx errors don't directly reject);

When request succeeds (resolve), parameter passed to then is an object: contains ok, status, statusText parameters; can parse response body by calling json(), text(), blob(), formData() methods, chain call returns to next then;

Chain call second then, callback function's parameter receives data parsed as JSON or text, can operate on this data;

```js
fetch('http://localhost', { method: 'GET' })
  .then((response) => {
    if (!response.ok) throw new Error(response.status);
    return response.json();
  })
  .then((data) => {
    console.log('Got data', data);
  })
  .catch((err) => {
    console.error('Failed to get data', err);
  });
```

### Fetch vs XHR Differences

**xhr**:

Compatible with IE7+ browsers;

Handles async results based on event listeners;

Native abort method to terminate request;

Cannot read response body in chunks during transmission; to handle large files, need server cooperation for chunked transmission, client manually concatenates;

Carries cookies by default;

Provides `onprogress` event, automatically triggers progress updates, can directly get loaded size and total size;

**fetch**:

Only supports modern browsers (Chrome 42+, Firefox 39+, Safari 10.1+);

Based on Promise, supports async/await, chain call then;

Needs browser native `AbortController` to generate signal (`signal`), associate with request then cancel via `controller.abort()`;

```js
// Create controller and signal
const controller = new AbortController();
const signal = controller.signal;

// Pass signal to fetch config
fetch('/api/long-task', { signal })
  .then((res) => res.json())
  .then((data) => console.log(data))
  .catch((error) => {
    // Canceling request triggers AbortError
    if (error.name === 'AbortError') {
      console.log('Request cancelled');
    } else {
      console.error('Error:', error);
    }
  });

setTimeout(() => {
  controller.abort(); // Cancel request via controller
}, 3000);
```

`fetch`'s `Response.body` is native `ReadableStream` object, can read response data segment by segment;

`fetch` by default doesn't carry cookies in cross-origin requests, need to manually set `credentials: 'include'` to carry (same-origin requests carry by default);

Need to convert response body to `ReadableStream`, manually read stream data and accumulate loaded size, calculate progress;

```js
fetch('/api/large-file')
  .then((res) => {
    const contentLength = Number(res.headers.get('Content-Length')); // Total size
    let loaded = 0; // Loaded size

    // Get stream reader
    const reader = res.body.getReader();

    // Recursively read stream data
    function readStream() {
      return reader.read().then(({ done, value }) => {
        if (done) {
          console.log('Download complete');
          return;
        }
        // Accumulate loaded size, calculate progress
        loaded += value.byteLength;
        const progress = (loaded / contentLength) * 100;
        console.log(`Download progress: ${progress.toFixed(2)}%`);

        return readStream(); // Continue reading next segment
      });
    }

    return readStream();
  })
  .catch((error) => console.error('Download failed:', error));
```

### Axios

Promise-based network request library, can automatically convert data to JSON format;

**Browser**:

- Based on `XMLHttpRequest` wrapper; wraps XHR's `onload`, `onerror`, `onabort` etc. events as Promise's `resolve`/`reject`;
- Simplifies XHR's scattered operations like `open()`, `setRequestHeader()`, `send()` via config object (like `method`, `headers`, `timeout`);
- Automatically converts data to JSON string when requesting, automatically parses JSON data when responding; uniformly handles network errors, timeout errors, HTTP errors (4xx/5xx status codes);

**Node.js Environment**:

- Based on http/https module wrapper;
- This part also follows Promise-ification and unified interface principles, converts callback-style APIs like `http.request()` to Promise, aligns with browser environment's config parameters (like `method`, `headers`, `timeout`);

**Install**: `pnpm i axios`

#### Making Requests:

```js
import axios from 'axios';

// Promise style
axios({
  method: 'post',
  url: 'https://frontest.sipc115.com/admin/auth/login',
  data: {
    userName: adminData.userName,
    password: adminData.password,
  },
}).then((res) => {
  console.log(res);
});

// try-catch style
try {
  const res = await axios({
    url: 'https://v1.hitokoto.cn/',
    method: 'get',
  });
  console.log(res);
} catch (error) {
  console.log(error);
}

// By request method
axios
  .get('/api/user', {
    params: {
      id: 1,
    },
  })
  .then((res) => {
    console.log(res.data);
  });
```

#### Request Config:

```js
axios({
  url: '/api/user',
  method: 'get',
  baseURL: 'https://api.example.com',
  timeout: 5000,
  params: {}, // URL query params ?a=1
  data: {}, // Request body (POST/PUT)
  headers: {
    Authorization: 'Bearer token',
  },
  withCredentials: true, // Whether to carry cookies
  responseType: 'json',
});
```

#### Axios Instance

Used to unify request config, avoid writing baseURL, headers every time;

```js
const service = axios.create({
  baseURL: '/api',
  timeout: 10000,
});

// Send GET request
service.get('/user');
```

#### Interceptors:

**Request Interceptor**: Executes before sending request; use receives two callback functions (only one will trigger):

First is callback executed before initiating request, parameter is request config; can validate, add token, modify parameters here, its return value becomes final config for HTTP request;

Second parameter is callback executed only when first callback throws exception or returns Promise.reject; if return is normal value passes to next Promise chain's then, if `throw error/Promise.reject(error)` passes to catch;

```js
axios.interceptors.request.use(
  (config) => {
    return config;
  },
  (error) => {
    return Promise.reject(error);
  }
);
```

**Response Interceptor**: Executes after server returns data, before processing request data; use receives two parameters: response function and error function; can uniformly handle business status codes here;

```js
axios.interceptors.request.use(
  (res) => {
    return res;
  },
  (error) => {
    return Promise.reject(error);
  }
);
```

# SSE

Server-Sent Events, a technology based on HTTP protocol for implementing server-to-client **unidirectional** real-time data push communication;

Server needs to declare `Content-Type: text/event-stream` via HTTP response header to start SSE stream, client can initiate connection and receive data via native `EventSource` API;

SSE uses HTTP long connection (`Connection: keep-alive`) mechanism; after connection established, doesn't immediately close, server can continuously push data to client through this connection until connection is actively closed or abnormally interrupted;

When connection disconnects due to network interruption, server failure, etc., `EventSource` automatically tries to reconnect (default reconnect interval is 3 seconds); server can customize reconnect interval via `retry` field; when reconnecting, client carries last received event ID via `Last-Event-ID` request header, server can resume data transmission based on this ID;

Server can only transmit text data, cannot directly support binary data (can convert to base64);

SSE has clear specifications for server-transmitted data format:

`event`: Optional, specifies event name (default event name is `message`), client can listen for specific types of messages via event name;

`data`: Required, data content (text format, usually JSON string), multi-line data needs each line to start with `data:`, finally ends with empty line `\n\n`;

`id`: Optional, event unique identifier, used for resuming data when reconnecting (client caches this ID, sends to server via `Last-Event-ID` when reconnecting);

`retry`: Optional, specifies client reconnect interval (milliseconds), overrides default 3 seconds;

```js
// Connect to server's SSE endpoint
const eventSource = new EventSource('http://127.0.0.1:80');

// Listen for default event
eventSource.onmessage = (event) => {
  console.log('Received data:', event.data);
};

// Listen for connection open
eventSource.onopen = () => {
  console.log('SSE connection opened');
};
```

### eventSource Instance

#### Properties:

**readyState**: Indicates current connection state, has 3 values:

0 (`EventSource.CONNECTING`): Connecting;

1 (`EventSource.OPEN`): Connection opened, can receive data;

2 (`EventSource.CLOSED`): Connection closed;

**url**: Returns complete URL of SSE server used when initializing `EventSource`;

**withCredentials**: Indicates whether to use credentials (like Cookies, HTTP authentication info) in cross-origin requests, default false;

#### Methods:

close(): Actively closes SSE connection;

#### Events

onopen: Triggered when SSE connection with server is successfully established (`readyState` becomes `1`);

onmessage: Triggered when receiving default event sent by server (when server doesn't specify `event` field, default event name is `message`); can get received data via parameter object's data property;

onerror: Triggered when connection error occurs (like network interruption, server returns non-200 status code), parameter is error object;

```js
eventSource.onmessage = (event) => {
  console.log('Received data:', event);
};

// Listen for connection open
eventSource.onopen = () => {
  console.log('SSE connection opened');
};

// Listen for connection error
eventSource.onerror = (error) => {
  console.error('SSE connection error:', error);
};
```

# WebSocket

**Message Push Methods:**

WebSocket: Client sends HTTP request containing `Upgrade: websocket` and `Connection: Upgrade` headers, server returns 101 status code indicating successful protocol upgrade, connection converts from HTTP to WebSocket; after connection established, client and server can send messages to each other at any time, no need to repeatedly establish connections, maintaining long connection state;

### Server

Install: `npm install ws`;

#### Create WebSocket Server

Use WebSocket.Server method, parameters are host, port, path (server access path, default /), verifyClient (method to get and verify client info)

```js
const ws = new WebSocket.Server({
  host: 'localhost',
  port: 3000,
  path: '/ws', // Server path is localhost:3000/ws
  verifyClient: (info) => {
    console.log(info);
    return true;
  },
});
```

#### ws Instance Methods

**ws.on**: Listens for specific events on server, first parameter is event name (connection, close, error), second parameter is callback function;

connection callback parameters: WebSocket instance object, this object represents connection established with client; through this object, you can send, receive messages, close connection, etc. with corresponding client;

WebSocket instance object's properties and methods:

client.send: Send data to client;

client.close: Close current client's connection;

client.on: Listen for specific events (open, close, error, message);

```js
ws.on('connection', (client) => {
  console.log('New client connected');
  // Triggered when receiving client message
  client.on('message', (message) => {
    console.log(`Received message: ${message}`);
  });

  // Triggered when client disconnects
  client.on('close', () => {
    console.log('Client disconnected');
  });

  // Triggered when error occurs
  client.on('error', (error) => {
    console.error('WebSocket error:', error);
  });
});
```

**ws.close**: Closes entire WebSocket server, terminates all current connections and stops accepting new connections;

```js
ws.close();
```

**ws.clients**: Gets Set of all currently connected WebSocket instances;

```js
ws.on('connection', (client) => {
  client.on('message', (message) => {
    console.log(`Received message: ${message}`);
    ws.clients.forEach((clients) => {
      if (clients.readyState === WebSocket.OPEN) {
        clients.send(message);
      }
    });
  });
});
```

### Client

For client, WebSocket is JS built-in object, used to establish persistent bidirectional communication connection between client and server, implementing real-time data transmission;

#### Create WebSocket Object:

```js
const socket = new WebSocket('ws://localhost:8080');
```

#### socket Instance

**socket.send**: Sends data to server, data can be string, Blob, ArrayBuffer or ArrayBufferView;

**socket.close**: Closes current WebSocket connection, first parameter is status code (like 1000 means normal close, default 1005 means no status code), second parameter is close reason string;

```js
socket.close(1000, 'Client actively closing');
```

**socket Events**

```js
socket.onopen = function () {
  console.log('Connection established, can start communicating');
};

socket.onmessage = function (event) {
  console.log('Received message:', event.data);
  // If message is JSON format, can parse to object
  const data = JSON.parse(event.data);
};

socket.onclose = function (event) {
  console.log(
    `Connection closed (status code: ${event.code}, reason: ${event.reason}`
  );
};

socket.onerror = function (error) {
  console.error('WebSocket error:', error);
};
```
