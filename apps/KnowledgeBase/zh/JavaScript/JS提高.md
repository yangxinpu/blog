# JS提高

### 事件循环

JavaScript 是单线程的，而浏览器是多线程的，网络请求、定时器、DOM 事件等能力由浏览器的其他线程负责执行；JavaScript 通过调用这些 Web API 发起任务后，不会阻塞主线程执行，当异步任务完成后，其回调会被放入任务队列中，事件循环机制负责在调用栈为空时，将这些回调取出并执行，从而实现 JavaScript 与浏览器多线程能力的协作；

JS为什么不设计成多线程：避免多线程带来的复杂并发问题（数据竞争、死锁），如多个线程同时操作 DOM，会导致状态不一致；

- 事件循环的本质是：在单线程执行栈与多个任务队列之间进行调度，使 JavaScript 能以非阻塞方式处理异步任务；
- 事件循环的执行顺序为：执行同步代码 → 清空微任务 → 执行一个宏任务 → 再清空微任务，循环往复；

#### 同步和异步

同步任务：在执行栈中按顺序执行；

异步任务：异步任务不会阻塞主线程，JS引擎在遇到异步操作时，会调用浏览器提供的 Web APIs（或 Node.js 中的 libuv）来处理异步操作，JS 引擎继续执行后续的同步代码，当异步任务完成后，其回调函数会被放入任务队列中，通过事件循环机制，在执行栈为空时被调度进入主线程执行；

任务队列分为微任务队列和宏任务队列：

- 宏任务：如script（整体脚本，第一个宏任务），网络请求（fetch）， `setTimeout`、`setInterval`、DOM事件，requestAnimationFrame等，每一轮事件循环只执行一个宏任务；

- 微任务：如 `Promise.then / catch / finally`，`queueMicrotask`， `MutationObserver`，会在当前同步任务执行完毕后立即执行，并且会被一次性清空，如果在执行微任务的过程中又产生了新的微任务，这些新的微任务会被放入当前微任务队列，并且会在这一轮中继续执行，直到微任务队列被彻底清空；

  ```
  setTimeout(() => {
    console.log("macro");
    Promise.resolve().then(() => {
      console.log("micro");
      Promise.resolve().then(() => {
          console.log("micro2");
      });
     });
    Promise.resolve().then(() => {
      console.log("micro3");
    });
  });
  //输出顺序为macro，micro，micro3，micro2（micro2是后面加入队列的微任务）


  queueMicrotask(() => {//标准微任务 API
      console.log('microtask');
  });

  //Node.js 中的微任务
  process.nextTick(() => {
    console.log('nextTick');
  });
  ```

#### 异步调度机制

在不阻塞主线程的前提下，把暂时无法立即完成的任务交给底层系统处理，并在合适的时间按既定优先级规则重新安排回主线程执行的机制；

- 当JS引擎遇到异步任务时，不会执行异步任务本身，而是交给浏览器的 Web APIs 或 Node.js 的 libuv在系统的底层处理（比如定时器在计时，网络在请求等）；
- 接着JS引擎继续执行同步任务；
- 异步任务本身完成后，对应的回调会被放入宏任务或微任务队列；
- 当执行栈为空时，事件循环会把任务的回调函数调度进执行栈（微任务队列要一次性清空，宏任务队列一次只取一个）

### 迭代器与生成器

#### 迭代器

迭代：从有序并且连续的目标源中，以逐个抽取的方式来访问数据；（遍历通常是基于迭代实现的）；

迭代器协议：必须实现next()方法，并且next方法返回{value,done}，符合迭代器协议的对象即迭代器；

可迭代对象：实现了 `Symbol.iterator` 方法的对象，比如数组，字符串，Set，Map；可以被for of和...扩展运算符遍历；

原生 `Object` 默认没有实现 `Symbol.iterator`，因此不是可迭代对象，不能被 `for...of` 遍历，可以手动实现 `Symbol.iterator` 使其变为可迭代对象；

```
let arr = [1,2,3];
console.log(arr[Symbol.iterator]().next());

//自定义对象的自定义的迭代器
let obj = {
    a:1,
    b:2,
    c:3,
    [Symbol.iterator](){
        var index =0;
        let map = new Map([["a",1],["b",2],["c",3]]);
        return{
        //返回next()方法
            next(){
                var mapEntries = [...map.entries()];
                if(index<map.size){
                    return {
                        value:mapEntries[index++],
                        done:false
                    }
                }
                return {value:undefined,done:true};
            }
        }
    }
}

for(let i of obj){
    console.log(i);
}
```

#### 生成器

Generator 是一种可以暂停执行并在之后恢复执行的函数，通过 `function*` 定义，调用后返回一个生成器对象；

生成器对象：

- next方法：启动或恢复生成器执行，执行到下一个 `yield / return`；
- return方法：用于终止生成器，返回`{ value: value, done: true }`；
- throw方法：向生成器内部抛出异常

生成器中：

- 调用一次next，执行到一个 yield/return，并且yield/return的值作为，迭代器返回对象的value；
- `next(value)` 的参数，会作为上一个 yield 表达式的返回值
- 如果调用过程遇到return，返回`{ value: return的值, done: true }`，后面的所有next都返回`{ value: undefined, done: true }`；

```
function* gen() {
  const a = yield 1
  console.log('a:', a)

  const b = yield 2
  console.log('b:', b)

  return 3
}
//获得生成器对象
const g = gen()
g.next()//返回{ value: 1, done: false }，停在：yield 1，a 还没赋值
g.next("X") //"X" 会赋值给a，输出'x'，执行yield 2，返回{ value: 2, done: false }
g.next("Y") //"Y" 会赋值给b，输出'Y'，执行return 3，返回{ value: 3, done: true }
//之后调用g.next都是{ value: undefined, done: true }
g.return(100)   // { value: 100, done: true }//可以直接终止生成器



//使用生成器生成迭代器
let obj = {
    a:1,
    b:2,
    c:3,
    [Symbol.iterator]:function* (){
        var index =0;
        let map = new Map([["a",1],["b",2],["c",3]]);
        var mapEntries = [...map.entries()];
        while(index < mapEntries.length){
            yield mapEntries[index++];
        }
    }
}
for(let i of obj){
    console.log(i);
}
```

### ES Modules模块

一个模块代表一个 独立的 JavaScript 文件，这个文件内部可以定义变量、函数或类，然后通过导出暴露给其他文件使用；

- 静态结构：模块的依赖关系在代码解析阶段就已经确定（ESM 的 import 必须写在模块顶层，不能放在函数内部，不能写在条件语句里，不能动态拼接路径）；
- ES 模块只会执行一次，当一个模块被多个地方导入时，JavaScript 只会加载一次模块，之后所有导入都会共享同一个模块实例；
- 模块作用域：默认是严格模式，this为undefined，模块内部声明的变量，不会挂到全局对象；
- `import` 得到的是被导出变量（包括普通变量）的实时绑定引用，不是值的拷贝，如果模块变量值改变，所有导入该模块的值也会改变（即模块顶部声明的变量会被共享同一个绑定）；
- 对于浏览器环境：只能通过http/https，data:，blob:协议加载ES模块，不能使用file://协议；（Node环境没有这个限制）；

模块加载流程：

- 解析模块：浏览器读取模块文件，解析 import 和 export，建立模块依赖图；
- 实例化模块：为每个模块创建模块记录和导出绑定；
- 执行模块：按照依赖顺序执行模块（深度优先执行其依赖模块），并且将该模块保存到模块注册表

```
//命名导出
export const fn = function () {
    console.log('模块');
}
//导入
import { fn } from './jsmodule.js'

//默认导出
export  default  function () {
    console.log('模块1');
}
//导入
import  fn  from './jsmodule.js'

//别名导入
import {fn as Fn} from './jsmodule.js'
//命名空间导入
import * as Fn from './jsmodule.js'//将导出的内容收集到一个Fn对象中
```

#### 动态模块加载

允许运行时在局部作用域按需加载模块；动态导入会返回一个 Promise；

- 不在首屏加载时引入，而是在需要时才加载；
- 可以在任意位置使用，支持条件加载；
- 支持代码分割：打包工具（如 Webpack / Vite）会对动态模块拆分成独立 chunk；

```
if (1) {
    import('./jsmodule.js').then(module => {
        module.default();//默认导出的对象
        module.fn();
    })
}
```

### JS运行时模型

```
//JS 执行代码的核心流程

1. 创建执行上下文（Execution Context）
2. 入栈（执行上下文栈）
3. 创建阶段（Creation Phase）
   ├── 创建词法环境（Lexical Environment）
   ├── 创建变量环境（Variable Environment）
   ├── 绑定 this
   ├── 建立作用域链（outer）
   └── 变量/函数提升（hoisting）
4. 执行阶段（Execution Phase）
   └── 逐行执行代码（变量查找、赋值等）
5. 出栈（执行结束）
```

#### 执行上下文

JS代码执行环境，定义了在代码执行时，变量、函数和对象如何被访问和管理；包括全局上下文和函数上下文；

执行上下文包含：

- 词法环境：用于存储 `let / const` 声明的变量，以及块级作用域；内部包含环境记录和外部引用（用于形成作用域链）；
- 变量环境：用于存储 `var` 声明的变量和函数声明；本质也是一个词法环境，但通常在执行过程中不会像词法环境那样频繁变化；
- `this` 对象：在全局上下文中，`this` 指向全局对象；在函数上下文中，`this` 指向调用该函数的对象；

相关机制

- 作用域链 ：作用域链是由多个词法环境 链接组成的，通常是当前执行上下文的词法环境加上外部环境的引用；
- 调用栈：在执行过程中，每当一个新的执行上下文被创建时，它会被压入调用栈。当该上下文执行完毕，栈顶的上下文会被弹出。

#### 词法环境

词法环境（ LexicalEnvironment）是 JavaScript 用来实现词法作用域的一种数据结构，由环境记录（Environment Record）和对外部词法环境的引用（outer）组成；

- 环境记录：用于存储当前作用域中的let/const变量、函数声明、函数参数等；有两种类型：
  1. 声明式环境记录：用“变量表”存变量，不是 JS 对象（不能通过 `this.xxx` 访问），如let/const声明的变量，函数参数等
  2. 对象式环境记录：用“对象”来当作用域，变量直接存到一个对象上，比如with声明的作用域，全局作用域（window）；
- 而 outer： 用于形成作用域链，实现变量的逐级查找；

```
{
  环境记录（Environment Record）: 存变量
  outer: 指向外层词法环境
}

function fn() {
  let a = 1;
}

fn Lexical Environment
   ├── a: 1
   └── outer → Global Lexical Environment
```

#### 变量环境

变量环境（Variable Environment）本质是一个词法环境（同样有环境记录和outer），用于存储 `var` 声明的变量和function声明的函数；

在执行上下文创建时`VariableEnvironment === LexicalEnvironment`，但当代码执行到块级作用域，才会创建新的 LexicalEnvironment；

特点：

- 只在“函数级作用域”生效，没有块级作用域；
- 会发生“变量提升”；
- 函数声明优先级更高；

```
VariableEnvironment = {
  EnvironmentRecord: {
    // 存 var 和 function
  },
  outer: 外部引用
}
```

#### VO/AO

变量对象（VO）是 ES5 及之前规范中用于描述执行上下文中变量存储的抽象模型，用于保存变量声明、函数声明和函数参数等标识符绑定（现在使用的是环境记录代替）；

- 在全局上下文中，变量对象通常表现为全局对象（浏览器中为 window）；
- 在函数上下文中，变量对象会被激活为活动对象（AO），此时会包含函数参数、arguments 对象、函数声明以及变量声明；

变量对象在创建阶段完成初始化（变量为 undefined，函数已赋值），在执行阶段会随着代码执行不断更新其值；

#### 作用域

作用域是指变量和函数可访问的范围，用于规定标识符的可见性；

JavaScript 的作用域 采用词法作用域（静态作用域），作用域在代码定义时就已经确定，与函数的调用位置无关；

- 包括全局作用域，函数作用域，块级作用域，模块作用域
- 作用域链是由多个词法环境通过 outer 引用连接形成的结构，用于在变量访问时进行查找
- 当访问一个变量时，会从当前作用域开始查找，如果未找到，则沿着作用域链逐级向上查找，直到全局作用域；若仍未找到，则返回 undefined 或抛出错误；

### 垃圾回收机制（GC）

垃圾回收是指：在堆内存中删除不被其他对象使用或无法访问的对象的过程，如果一个对象无法从根对象（全局变量）访问到，他就会被回收；

- JS引擎会周期性触发垃圾回收机制，并且当系统可用内存较低时会触发得更频繁；

- 强引用所指的对象是否被回收取决于 GC 的可达性分析：如果闭包、事件回调、定时器等仍然持有对某些对象的引用，并且这些引用链可以从根对象访问到，那么这些对象就不会被垃圾回收，从而可能导致内存泄漏；

- WeakMap 和 WeakSet 使用弱引用，不会影响被存储的对象的可达性判断；当对象只被弱引用持有，不被其他强引用对象只有，就会被垃圾回收；

  ```
  let obj = {};
  let wm = new WeakMap();

  wm.set(obj, "data");//GC 在判断 obj 是否可达时，不会把 WeakMap 这条引用算进去
  ```

垃圾回收方法：

现代 JavaScript 引擎主要采用基于标记清除的垃圾回收算法，并结合分代回收、增量标记等优化手段，以提高回收效率并减少对主线程的阻塞；

- 标记清除法：从根对象出发，遍历并标记所有可达对象；然后再遍历堆内存，清除所有未被标记的对象。该算法基于可达性判断，可以有效解决循环引用问题；
- 引用计数法：为每个对象维护一个引用计数，当有新的引用指向该对象时计数加一，引用失效时计数减一；当计数为 0 时，对象会被立即回收。但该算法无法处理循环引用问题；

### 浏览器渲染原理

#### 浏览器内核

**webkit**：（苹果）核心为模块化渲染，主要由WebCore和JSCore组成；WebCore负责解析HTML和CSS（支持现代Web标准，如FlexBox，WebGL，移动端等），构建DOM树和CSSOM树，并通过布局和绘制生成最终页面；JSCore负责即时编译JS代码（JIT），支持ES6+，WebAssenbly；

应用的浏览器：Safari（苹果的Mac，iPhone，iPad）；

**Blink+V8**：（谷歌）Blink是根据webkit的WebCore分支开发的，核心作用是解析HTML和CSS渲染页面，其采用多线程设置（渲染进程和浏览器主进程分离）；V8引擎，负责JS即时编译，当Blink解析到script标签时会调用V8的编译接口进行JS编译；

应用的浏览器：Chrome，Edge，Opera；

#### 进程和线程

**进程：**进程是操作系统分配资源（如内存、CPU 时间等）的基本单位，是一个程序的执行实例；比如开一个浏览器就是启动一个浏览器进程。打开一款app就是打开一个进程；

浏览器是多进程的，有一个主进程，GPU进程，第三方插件进程，每打开一个页面都会新开一个进程（浏览器内核渲染进程）；

**线程：**线程是进程内的执行最小单元，一个进程可以包含多个线程，它们共享进程的资源；

在浏览器渲染进程（内核）中，包含多个线程：

GUI线程：负责渲染浏览器页面，解析HTML，CSS，构建渲染树，布局和绘制等；

JS引擎线程：负责解析并执行JS代码的主线程，JS引擎会一直等待任务队列中任务的到来，然后处理JS代码；

事件触发线程：主要用于控制事件触发，当事件触发时，就会把事件回调函数推进事件队列，等待JS引擎线程执行；

定时器触发线程：主要控制setInterval和setTimeout，计时完毕后就会把定时器的回调函数推进任务队列中，等待JS引擎线程执行；

异步http请求线程：监测http请求的状态，当状态发生改变，则将对应的回调函数推进任务队列中，等待JS引擎线程执行；

注意：微任务不会单独开启线程，而是由JS引擎线程在特定时机统一处理，当主线程执行完当前同步代码后，会先清空微任务队列，再去检查宏任务队列（如`setTimeout`、`DOM事件`等）；

GUI线程和JS引擎线程是互斥的，不能同时执行（由于JS可以操作DOM，如果在修改元素的同时渲染页面，会导致前后渲染结果不一致，为了防止这种结果，当JS引擎执行时，GUI线程会被挂起，等待JS引擎空闲时才被执行）；

#### 浏览器的渲染原理

当浏览器接收到一个HTML文件后，会产生一个渲染进程任务，并将其传递给主线程的消息队列，在事件循环机制下，渲染主线程取得消息队列中的渲染任务，开启渲染流程；

**渲染第一步是解析HTML文件**：

浏览器在解析前会启动一个预解析的线程：提前扫描资源引用，提前发起网络请求下载外部的资源文件（如JS，CSS，图片，视频等），当然如果主线程在解析到style标签时，CSS文件还没有下载完成，不会在此等待而是继续解析后面的HTML，但是如果解析到script标签时，JS文件没有下载完成，主线程则会等待JS文件的下载并解析完JS代码后才继续解析后面的HTML（因为JS操作DOM可能改变初始DOM树，JS代码会在初始DOM树构建完后执行，再生成DOM树）；

解析完HTML后会产生DOM树（所有形成的DOM节点树形结构）和CSSOM树（解析CSS后生成的树形结构），浏览器将CSSOM树形上的样式规则映射到DOM树上的DOM节点后生成渲染树（渲染树上仅包含需要显示的元素，不包括display设置为none的节点）；

**渲染第二步是布局（重排）**，根据渲染树上每个节点的几何信息（如元素尺寸，边距边框，层级关系

（z-index），浮动，定位）计算元素位置生成布局树；

**渲染第三步是绘制（重绘）**，获得布局树后，根据布局树进行分层（即将页面内容划分为多个独立的图层，当修改图层内容时，只需重绘该图层），浏览器再将每个图层进行分块（如，一个高度为 2000px 的图层，可能被分成 8 个 256px 高的块），并对每个块进行光栅化（将每个矢量描述转化为屏幕的像素点），生成像素数据；

**渲染第四步是合成：**浏览器将所有独立的图层的像素数据通过GUP按照视觉层级合并，并通过显示系统输出到屏幕上；

#### 前端代码

把 JavaScript 源代码从字符流转换为可执行语义结构（AST + 作用域信息）；

- 词法分析器（Lexer / Tokenizer）：将源代码字符流变成Token流（具有语义的最小语法单元，如关键字，标识符，运算符等）；
- 语法分析器：将Token 流转化为AST（抽象语法树，保留语义结构），并做语法合法性校验，作用域信息收集，严格模式判断；
- 最终输出AST，作用域与变量绑定信息，语法模式（严格 / 非严格），可供字节码生成使用的结构化表示；

#### 执行流水线

执行流水线负责把语义结构（AST）逐步转化为机器代码；执行流水线在运行时核心提供的语义与状态之上工作

- 字节码生成器：将AST + 作用域信息变成平台无关字节码（JS引擎为了解释执行和 JIT 编译而设计的一种中间指令集）
- 解释器：顺序执行字节码指令（包括读取指令，解析操作数，修改寄存器 / 栈）， 维护执行上下文（函数，全局，eval），处理 JS 的动态语义， 调用运行时，处理控制流（trycatch，循环等），收集类型反馈给JIT；
- JIT 编译器：（V8 没有传统 Baseline JIT）把频繁执行的字节码转化为更少检查、更少跳转、更少抽象的 CPU 机器码，即执行热点代码，让其接近原生速度，和解释器互补；
