# React Basics

react package: React core library, defines the UI abstraction model

- JSX → `React.createElement` (generates virtual DOM / React Element)
- Component model (Function components / Class components)
- Hooks (`useState`, `useEffect`, etc.)

React Renderer: The execution layer that converts UI described by React (React Element / Virtual DOM) into specific platform views, responsible for building Fiber tree, diff between old and new Fiber trees, and executing responsive update operations;

- Web → `react-dom`
- Native → `react-native`
- SSR → `react-dom/server`

createRoot method: Receives a DOM as parameter, creates a Fiber Root (root Fiber node)

render method: A method provided by the root container object created by createRoot(), receives a React component's virtual DOM as parameter, submits an update to Fiber Root, triggering scheduling and rendering flow;

```jsx
createRoot(document.getElementById('root')).render(
  <StrictMode>
      <App />
  </StrictMode>
)
```

### JSX Syntax

Uses HTML-like syntax to describe UI structure, which is ultimately compiled into JavaScript; (e.g., `const element = <h1>Hello</h1>` will be compiled to `const element = React.createElement('h1', null, 'Hello')`)

- Must have a root node, can use Fragment (doesn't generate extra DOM);
- Tag names starting with lowercase → treated as native DOM tags, tag names starting with uppercase → treated as React components (variable references);

#### Interpolation Syntax

Use { }, inside { } you can write any JS expression (function calls, ternary operators, variables, etc., but not statements);

- When iterating over tags, each tag must have a unique key;
- false / null will not render;

```jsx
  <div>
    {
      ["Vue", "React", "Angular"].map((item) => {
        return <div key={item}>{item}</div>
      })
    }
  </div>
```

- Style class names should use className;

- Inline style must be an object;


#### styled-components

A CSS-in-JS solution in the React ecosystem that allows you to write CSS directly in JavaScript/TypeScript and bind styles to components;

- styled-components is essentially calling a function (tag function), passing CSS code in template strings as parameters, and generating a unique hash class name for each style component (like `sc-bdVaJa`); styled-components creates a `<style>` tag at runtime (or reuses existing ones) and inserts it into `<head>`, writing the generated class name and corresponding CSS rules into the `<style>` tag;

Install: `npm install styled-components`;

##### Basic Syntax

- `styled-components` supports using interpolation expressions `${}` in template strings, interpolation can be: JS variables, functions (receives `props`, returns CSS values)

- `&` represents the class generated by the current component

- `styled(existing component)` is used to extend existing styles, similar to CSS inheritance + override;

```js
import styled from "styled-components";

const BaseButton = styled.button`
    font-size: 20px;
    padding: 10px;
    border-radius: 10px;
`

const PrimaryButton = styled(BaseButton)`
    border:1px solid ${props => props.color || 'black'};
    color:${props => props.color || 'black'};
    &:hover{
        background-color:${props => props.color || 'black'};
        color:white;
    }
`

const Home = () => {
    return (
        <>
            <h1>Home</h1>
            <PrimaryButton color="blue">Primary Button</PrimaryButton>
        </>
    );
}

export default Home
```

### Functional Components

The function itself serves as the render function, receives props, and the return value is the rendered component;

Hook: Used to introduce state management, side effect handling, and logic reuse capabilities (custom hooks) in functional components, essentially a state storage and side effect scheduling system based on call order, mounted and running on corresponding Fiber nodes;

- The functional component itself destroys local variables after execution, unable to maintain state across multiple renders, while the Hook chain moves state out of function scope and stores it on corresponding Fiber nodes;
- The head node of the Hook chain is saved on the current component Fiber's `memoizedState`; and each Hook node saves its own state information, such as: current state value, update queue, side effect dependencies, next Hook node pointer;
- React's Hook state reuse relies on call order, not variable names. When React re-renders a functional component, it takes out corresponding nodes from the old Hook chain in the order of Hook calls, and builds a new Hook chain based on this (a new Hook chain is created for each update), thus achieving state reuse (therefore, Hooks must be called at the top level of the component in a fixed order, and cannot be written in loops, conditions, or nested functions, otherwise it will cause order confusion);

Update Queue: All updates produced by setState and dispatch don't immediately modify state, but create an update object and put it into the corresponding Hook node's update queue, which is processed uniformly by React in subsequent scheduling;

#### props

props is data passed from parent component to child component, passed to child component function's parameter in object form;

- Child component can use destructuring assignment and set default values when receiving parameters;
- props is a read-only object and cannot be modified;
- Parent component can achieve child-to-parent communication by passing function references;

```js
function App() {
  return <User name="Alice" age={25} />;
}

function User(props) {
  return (
    <div>
      <h1>{props.name}</h1>
      <p>{props.age}</p>
    </div>
  );
}
```

#### useState

Used to add state in functional components; parameter can be initial value of state variable, or an initialization function; returns an array of state variable and state update function;

- Initialization function: This function is only called during first render to get the initial value, and won't be executed during subsequent component re-renders, but the function itself will be recreated as the component executes;
- Initialization function must be a pure function, cannot have side effects, avoid writing as external function, cannot depend on external state;
- State update function: Used to submit state change requests to React, thus triggering component re-render; it can receive a new value, or a callback function with old state as parameter; calling state update function doesn't immediately modify state, but React batches multiple calls in the current update cycle and then updates and renders uniformly; therefore, state update function calls execute synchronously, but state updates themselves are delayed, behaving like async;

In React, multiple setState calls (including nested calls) are put into corresponding Hook's update queue and executed uniformly in the same batching context, thus only triggering one render;

React 18 introduced automatic batching mechanism, making batching happen not only in event handlers but also in async callbacks (like setTimeout, Promise), only when updates cross different task boundaries (like after await) will multiple renders occur;

```jsx
import { useState } from "react";

const Home = () => {
    const [count, setCount] = useState(0);
    const [count,setCount] = useState(()=>handleCount());
    const [count,setCount] = useState(handleCount());//handleCount() is called every time component renders
    return (
        <>
            <div onClick={() => setCount(count + 1)}>Button{ count }</div>
        </>
    );
};
```

#### useEffect

Used to handle operations with side effects in functional components;

Side effect function: During function execution, besides returning UI or calculated values, extra impacts on external environment or operations depending on external environment, such as DOM manipulation, data requests, event listeners, modifying localStorage/sessionStorage, printing logs (`console.log`), etc.; should not use setState in side effect function, otherwise may cause infinite rendering

```text
render → commit → cleanup → effect
```

- Executes side effect function asynchronously after component mounts (i.e., after component renders DOM and completes browser painting);
- When dependencies change (if dependencies don't change, side effect function won't execute), component will re-render first (calculate changes) and commit updates;
- Then before executing new side effect function, execute previous cleanup function;
- When component unmounts, all `useEffect` cleanup functions will execute, but side effect functions won't execute again;

Parameters and Return Value:

- First parameter is side effect function;
- Second parameter is dependency array (when empty array, only called once during initial render, when not written, called every time component triggers re-render);
- Return value is side effect cleanup function;

useEffect closure problem:

Side effect function captures variable values in current render cycle (forming closure), when there are async operations in side effect, even if component subsequently re-renders and state changes, async callback still accesses old values from previous render cycle, thus causing the problem of reading stale state;

Solutions include correctly declaring dependency array to let effect re-execute (old async will be cleaned up), using `useRef` to hold latest value, or using `useEffectEvent` to avoid stale value problems caused by closures

```jsx
import React, { useState,useEffect } from "react";
import './home.css';
const Home = () => {
    const [count, setCount] = useState(0);
    //Simulate mount
    useEffect(() => {
        console.log("componentDidMount");
    }, [])

    //Simulate update
    useEffect(() => {
        console.log("componentDidUpdate");
    }, [count]);

    //Simulate unmount
    useEffect(() => {
        return () => {
            console.log("componentWillUnmount");
        }
    }, [])
    
    return (
        <>
            <div onClick={() => setCount(count + 1)}>Button{ count }</div>
        </>
    );
};
```

#### useEffectEvent

`useEffectEvent` is used to solve stale state problems caused by closures in `useEffect`, it allows callbacks in side effects to always access latest state and props when executing, while not triggering `useEffect` re-execution due to changes in these values, thus avoiding repeated execution from adding them to dependency array;

The essence of `useEffectEvent` can be understood as React internally encapsulating a mechanism similar to `useRef`: during each component render, saves latest state and props to internal reference container, and returns a reference-stable callback function; this callback function reads latest values from this container when executing, thus ensuring callback reference doesn't change while avoiding closure capturing stale state;

```jsx
import { useState, useEffect, useEffectEvent } from "react";

export default function App() {
  const [count, setCount] = useState(0);
  const logCount = useEffectEvent(() => {//Get latest state and execute corresponding operation
    console.log("count:", count);
  });
  useEffect(() => {
    const timeoutId = setTimeout(() => {
      logCount();
    },3000);
    return () => {
      clearTimeout(timeoutId);
    };
  }, [logCount]);
  return (
    <>
      <div>App</div>
      <div>
        <button onClick={() => setCount(count + 1)}>
          click me
        </button>
        <div>count: {count}</div>
      </div>
    </>
  );
}
```

#### useLayoutEffect

`useLayoutEffect` is used to **synchronously** handle layout-related side effects, its usage is basically the same as `useEffect`, but execution timing is synchronous after DOM update completes, before browser painting,

Used for reading DOM information and manipulating DOM, but may also block browser painting, affecting performance;

```jsx
import { useRef, useState, useLayoutEffect} from 'react';

export default function App() {
  const ref = useRef(null);
  const [count, setCount] = useState(0);

  useLayoutEffect(() => {
    const animationId = requestAnimationFrame(() => {
      ref.current.style.transform = `translateX(${count}px)`;
      setCount(count + 1);
    });
    return () => cancelAnimationFrame(animationId);
  }, [count]);

  return <div ref={ref}>Centered Box</div>;
}
```

#### useRef

Essentially a container, internally stores data through `current` property, used for:

- Cross-render data persistence: Created reference object remains unchanged throughout component lifecycle, and modifying its value won't trigger component re-render;
- Get DOM element: Operating DOM directly triggers rendering;

Cross-render data persistence uses:

- Save timer IDs (for clearing timers),
- Solve useEffect closure problems (get latest state),
- Store WebSocket, chart instances (ECharts/AntV), map instances that need cross-render reuse after creation, cannot create new ones every render;

```jsx
//Operate DOM
const Home = () => {
    const div = useRef(null);
    return (
        <>
            <div ref={div}>Button</div>
        </>
    );
};

//Cross-render data persistence
import { useRef, useEffect, useState } from 'react';
function ClosureFix() {
  const [count, setCount] = useState(0);
  // Use ref to store latest count (sync update every render)
  const countRef = useRef(count);

  // Every time count changes, sync to ref.current
  useEffect(() => {
    countRef.current = count;
  }, [count]);

  useEffect(() => {
    // Async task: print latest count after 1 second
    setTimeout(() => {
      console.log('Latest count:', countRef.current); // Access ref, get latest value
    }, 1000);
  }, []); 

  return <button onClick={() => setCount(prev => prev + 1)}>count+1</button>;
}
```

#### useContext

Used for cross-component state sharing, allows components to directly access and modify global or cross-level shared state, usage flow:

- First use createContext to create Context;
- Then provide Context data in parent component;
- Finally use useContext in child component to consume Context;

useContext depends on the Context state container object with identifier created by createContext, internally contains three key parts:

- Provider component: Used to inject shared data into component tree, will write current `value` into Context's current value (_currentValue) and push onto context stack, so that its subtree can read this value during rendering;
- Consumer component: When `useContext` renders, it directly reads Context's current value (set by nearest Provider during rendering);
- _currentValue property: Internal field mounted on Context object, used to store current value of this Context in current rendering environment;

Principle:

- When child component calls `useContext(Context)`, React reads current value from nearest upper `Provider` from Context stack during current component's rendering process;
- At the same time, React records its dependency on this Context on the corresponding Fiber of current consumer;
- When a `Provider`'s `value` changes, React propagates updates from this Provider's subtree, finds consumer Fibers that depend on this Context, and schedules these components to re-render;

  ```html
  <A.Provider value={1}>
    <B.Provider value={2}>
      <Child />
    </B.Provider>
  </A.Provider>
  
  Enter A → Set value = 1
    Enter B → Set value = 2
      Render Child (reads 2)
    Exit B → Restore value = 1
  Exit A → Restore default value
  ```

Notes:

- When value is an object, new reference is generated every time parent component renders, causing all consuming child components to re-render, can use useMemo to cache value, only update reference when dependencies change;

```jsx
//In src/context/ThemeContext.js
import { createContext } from 'react';
const ThemeContext = createContext('light');
export default ThemeContext;

//In parent component
import { useState } from 'react';
import ThemeContext from './contexts/ThemeContext';
import Home from './components/home';
const App = () => {
  const [theme, setTheme] = useState("light");
  return (
    <div className="App">
      <ThemeContext.Provider value={{theme, setTheme}}>
        <Home />
      </ThemeContext.Provider>
    </div>
  );
};
export default App;

//In child component
import { useContext } from "react";
import ThemeContext from "../contexts/ThemeContext";
import './home.css';
const Home = () => {
    const {theme, setTheme} = useContext(ThemeContext);
    return (
        <>
            <h1 style={{color: theme === "dark" ? "white" : "black"}}>Home</h1>
            <button onClick={() => setTheme("dark")}>dark</button>
            <button onClick={() => setTheme("light")}>light</button>
        </>
    );
};

export default Home;
```

#### useCallback

Problem solved: Every time functional component renders, internally defined functions are recreated (generating new references). If this function is passed to child component as props, even if function logic hasn't changed at all, child component will think props changed (because reference changed), thus triggering child component re-render, causing performance waste;

Using useCallback can cache function references, only recreate function when dependencies change, avoiding unnecessary re-renders of child components caused by frequently updated functions passed from parent component (can be used with memo);

useCallback parameters and return value:

- First parameter is the function to be cached;
- Second parameter is dependency array;
- Returns cached function reference, can call function through this reference later;

```js
  const callback = useCallback(() => {
    console.log('callback');
  }, [])
```

#### memo

Used to declare that child component (or component module) should cache render results, avoiding unnecessary re-renders of child components when props haven't changed substantially;
